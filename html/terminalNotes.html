    • Terminal for Windows
      • Open VS Code, select terminal, and new terminal.
      • You can download and install a Git Bash app
    • Navigating inside the terminal (changes nothing in the code)
      • pwd - prints out the path that you are currently in / print working directory (cd - windows)
      • ls - lists the contents of a folder / of a current directiry (dlr - windows)
      • ls -a - lists all the files, including hidden files (dlr /A - windows)
      • ls <folder> - lists the contents of a specific folder (dlr - windows)
      • ls <folder1>/</folder2> - we can chain folders together to see waht's inside the folder1 (dir <folder> - Windows)
      • cd <folder> - changes the directory (as if you double-clicked on a folder).
      • cd .. - goes one level back / one folder
      • open . - opens the current folder or directory in Finder (start . - windows)
      • open <folder> - opens folder in Finder (start <folder> - windows)
      • clear - clears the Terminal
      • code . - opens in VS code, you need to connect the terminal to VS code (LINK)
    • Creating and deleting folders
      • touch index.js - creates a file witht he name index.js in the directory you are currently in (ni file.js - Windows)
      • touch index.js styles.css - creates multiple files (ni file.js file.css - Windows)
      • touch Oldagram/index.html - creates a file in a specific directory without being in that directory
      • mkdir unit-converter - creates a folder
      • mkdir Projects/Scrimba/solo-project-1 - createa folder in a specific directory
      • rm index.js - deletes a file
      • rm Oldagram.index.js - deletes a specific file inside a folder
      • rm -rf <folder> - deletes a folder
      • rm - rf Developer/old-project - deletes a folder in a specific directory (r - recursive, f - force)
      • You don't have to create and delete files and folders in the terminal. normally, when you are working on a project, you will be managing them in VS code.

    • Git
      • Git is a version control system used for tracking changes in computer files. It is generally used for source code management in software development.
      • Git creates checkpoints as well, you can check the history and get back to the eirlier version of it. It also allows other people to join you in editing the project at the same time as you.
      • git status - checks the status of the repo
      • git init - initialises a repo of a folder we are currently in; we do it only once per project. Make sure you are in the correct folder before initialising a repo. If you cloned a repo from GitHub, you probably won't need to initialize it.
      • git add . - adds the changes to the staging area
      • git add file1.js file2..html file3.css - adds the selected files to the staging area
      • git restore --staged file3.css - unstages a file / puts it back to the working directory
      • git commit -m "Your message" - makes a commit of the sataged files
      • git push or git push origin main - pushes the project to the GitHub repo.
      • git log or git log --oneline - shows the history of the commits
      • Goes from working directory (This is your working space, uou create, delete, modify files) to (git add) staging (here you have the files that you wuold like to commit.) area (index) to (git commit) repository (This is where the history of your project is stored).
      • Before running git init, run git status to verify that you are not currently inside of a repo.

    • Set Up Your Git
      • Name: type your name or any name that you want git config --global user.name "Your Name"
      • To check your name: git config user.name
      • Email: it's better to use the same email as your GitHub account git config --global user.email your@email.com
      • To check your email: git config user.email
      • These can be changed later as well, but don't do it all the time.
    • Cloning
      • git clone url - copy the url of the repo that you created on GitHub and run this command.
      • By running this command it will create a folder with this repo in your current working directory, so make sure to navigate into the disired folder before running it.
      • before cloning run git status to verify you are not currently in a repo.
      • After cloning, you will have a copy of the repo on your local machine, and you can start working on it.
    • Connect GitHub repo with VS Code on your local machine
      • Create a repo in your GitHub account.
      • Copy the URL of your repository.
      • Open VS Code - Explorer - Clone Repository. Use clone when you don't have a copy of the repository on your local machine.
      • Paste the copied URL
      • Choose whhere you will store it locally. Make sure not to use spaces in your folder and file names. Use - or _ instead.
      • Now when you have your repo on VS Codce, make any changes in your files, so we can see them when we push the code up to GitHub.
    • Push code to GitHub
      • The structure is working directory (where you create, delete, or modify files.), use git add, then your in the staging area (Where you have the files you would like to commit), use git commit, then your in your repository (Where the history of your project is stored).
      • Open terminal.
      • Save all changes to your code.
      • Make sure you are in the correct folder. To check that, run pwd in the Terminal and it will show where you are right now. Use cd foldername to enter a folder or cd .. to go one folder back.
      • git init - You cannot make use of git in the project if you don't initialize/start it first with the git init command. when you run this command in your terminal, you will also see a U symbol, which means your files are untracked. You only run this command onoce per project. You won't be able to use any git commands without initialising git first.
      • when you clone an existing repository from GitHub to your local machine, you probably already have git initialised and will not need to run git init command. But to check if you have it initialised or not, open the Terminal, make sure you are in the folder that contains the project that you are working on, run git status. You will see a message and if it is “fatal: not a git repository”, it is not initialised yet and you need to run git init command.
      • git add . (the dot means that there are unsaved changes) - Adding the project to the staging area helps Git track your project and see changes you have made to it. To add your project to the staging area, run the git add . command. When you run this command, you will see that the U symbol automatically changes to A. This means that your files have been added to the staging area and are now being tracked by Git, waiting to be committed.
      • git commit -m "Your message here" - By running the git commit command, you are saving this particular stage and the changes to your project permanently in the Git repository. And although you might make few changes to the project in the future and commit them as well, this particular commit you made now will still be saved in the Git repository and can be accessed anytime. So after your project has been added to the staging area, the next thing you'll want to do is commit it by using the git commit -m “commit message" command. When you run this command, you should notice that the A symbol in the project is no longer there, and the project looks as you'd expect once again.
      • git push - Finally push to your GitHub repository

    Represented by U in terminal.

    A commit should encopasss a single feature, change, or fix. This makes it easier to undo or rollback changes later on. It also makes your code or project easier to review.
